# Shared-SDK Abstraction Review Rubric — 6축

공유 라이브러리(SDK)로 끌어올리는 **공개 추상화**(port·SPI·DTO·autoconfig·공개 타입)가 *올바른 공용 추상화인가*를 보는 채점표. `abstraction-critic` 에이전트가 이 rubric을 실행한다.

- **무엇을 보나**: "이 추상화가 공유 SDK로서 옳게 그어졌나" — 설계 *seam*·중립성·공통성. 일반 버그·스타일·런타임 정확성은 범위 외(그건 코드리뷰·스택별 reviewer 몫).
- **언제 쓰나**: 가장 큰 레버리지는 **코드 작성 전(설계 게이트)** — 추상화의 치명적 실수는 "애초에 이렇게 추상화하면 안 됐다"라서 PR 코드리뷰로는 늦다. 기존 모듈 사후 감사에도 쓴다.
- **버전**: v1.0 (cross-module 감사에서 반복 검출된 패턴을 운영 체크로 환원해 도출).

---

## ⚠️ 먼저 — rubric의 한계

이 rubric은 **"하한선 보장 도구"**이지 **"설계 종합 점수"**가 아니다.

- 판정 나쁨 = *"이건 확실히 공용 추상화로 문제 있음"* (참)
- 판정 좋음 = *"기본 seam은 갖췄음"* (실효·도메인 정합은 별개)

rubric이 **못하는 것**: 추상화의 실제 효용 · 미래 소비자 수요 예측 · 도메인 적합성. → **사람 판단·경험적 공통성 조사(decision-researcher)와 결합**해야 의미.

경험적 공통성(축 1)은 한 레포만 봐선 판단 불가 — **서버교차 스캔이 필요**하므로 `decision-researcher`가 보조한다. 나머지 5축은 추상화 표면(코드/설계)만으로 분석 가능.

---

## 6축 정의

| # | 축 | 한 줄 정의 |
|---|---|---|
| 1 | **commonality** | 정말 ≥2 소비자가 공유하는 추상화인가, 한 앱 코드를 들어올린 것인가 |
| 2 | **seam** | 안 변하는 축과 변하는 축이 분리됐나 (가변 선택을 코어에 박았나) |
| 3 | **isp** | 한 인터페이스가 여러 역할을 융합했나 / 미사용 메서드·과다 표면 |
| 4 | **neutrality** | 이름·계약이 특정 구현 기술을 박았나 (구현 교체 시 공개 계약 파단) |
| 5 | **yagni** | 상상 케이스까지 과대 추상화 / 반대로 leaky·앱특화 과소 추상화 |
| 6 | **conventions** | 의존 방향·zero-config·레이어·"소비측이 인프라 제공" 원칙 준수 |

## 축별 운영 체크포인트 (flag 기준)

채점은 vibes가 아니라 아래의 *측정 가능한 신호*로 한다. 신호 적중 시 해당 축에 finding. 각 체크의 `= 심각도`는 **기본값**이며, 맥락이 분명하면 critic이 한 단계 조정할 수 있다(근거 명시).

### neutrality (N)
- **N1 — 구현 기술 토큰이 공개 식별자에 박힘.** port/spi/계약 패키지의 타입명·메서드명·빈명·DTO명에 *특정 구현 기술* 토큰이 등장하면 flag. 금지어 사전(예): `Redisson·Resilience4j·JPA·MySQL·Kafka·SQS·<벤더명>` + 도메인 인증 메커니즘명. → 구현 교체 시 공개 계약이 깨진다. = major.
  - **예외 — 범용 개념어.** `Queue·enqueue·Cache·Lock·Store·Publisher` 등 범용 자료구조/패턴 용어는 *그 자체*가 추상화의 본질이면 중립이다(예: 진짜 큐 추상화의 `Queue`). flag는 **범용어가 특정 트랜스포트/구현 선택을 비(非)해당 추상화에 새게 할 때**만 — 예: 트랜스포트가 큐로 고정될 이유가 없는 outbox relay 계약에 `Queue/enqueue`가 박혀 HTTP 등 타 채널을 배제하면 flag.
- **N2 — 포트가 구현 라이브러리 타입을 노출.** `toXxx()`로 구현 타입을 반환하거나, public 시그니처 파라미터/반환에 구현 패키지 타입을 직접 노출. = major.
- **N3 — Javadoc이 특정 구현/도메인을 유일 구현·예시로 못박음.** "Redis 기반", "Product-like", 특정 앱 도메인 예시 등. = minor.

### seam (S)
- **S1 — 불변 흐름 + ≥2 가변 축 융합.** claim→send→mark 같은 불변 절차에 저장소·트랜스포트(채널) 같은 가변축이 한 인터페이스로 묶이면 flag. = major.
- **S2 — 핵심 불변 기능이 선택적 기능 조건에 종속.** `@ConditionalOnBean/@ConditionalOnClass`가 "있으면 좋은" 의존(메트릭 등)에 "반드시 동작해야 할" 기능(로깅 등)을 묶음. = major.
- **S3 — 모듈 간 공유 계약이 코드 타입이 아닌 주석으로만 동기화.** 응답 포맷·MDC 키·헤더 규약을 공유 타입 없이 주석으로만 맞춤 → 조용한 불일치 위험. = major.

### commonality (C) — 경험적, decision-researcher 보조
- **C1 — 앱 베이스 패키지·골격명 절대경로 누수.** 소스/리소스에 소비 앱의 베이스 패키지·프로젝트 골격명·절대경로가 박힘. = major.
- **C2 — 사용자 대면 자연어 하드코딩 + MessageSource seam 부재.** 한 모듈 내 언어 혼재(예: 영/한)도 flag. = minor.
- **C3 — "현재 ≥2 소비자 실측 공유" 증거 유무.** 1소비자뿐이면 "수렴 대기"가 아니라 **"승격"**으로 분류하고, 도메인 특화 팩토리/필드가 SDK로 함께 끌려오는지 별도 평가. = minor (단, 도메인 특화 끌려옴이 확인되면 major).

### isp (I)
- **I1 — 미사용 SPI 메서드.** 소비측(타 모듈) 실호출 0건이고 어댑터 내부 자가소비뿐인 메서드. = minor.
- **I2 — god 인터페이스.** 단일 인터페이스 메서드 수가 임계(예: 8개) 초과 + 역할군 ≥2 혼재(매퍼+리포+트랜스포트 등). = major.

### yagni (Y)
- **Y1 — 실구현체가 테스트 스텁뿐인 계약.** 선언만 있고 실제 소비자가 없음. = minor.
- **Y2 — 계약-구현 불일치.** 계약이 선언한 동작을 구현이 이행하지 않음(예: 타입 파라미터 무시). = major.

### conventions (V)
- **V1 — @Component 단독 등록(스캔 의존).** SDK 라이브러리 빈이 `@AutoConfiguration + @ConditionalOnMissingBean` 대신 컴포넌트 스캔에 의존 → 스캔 범위 밖 소비자는 빈을 못 얻음. = major.
- **V2 — 스캔 부수효과 타입이 SDK 패키지에 노출.** `@Entity` 등 — 가시성 축소로는 바이트코드 스캔을 못 막는다. = major.
- **V3 — 발행한 공유 규칙을 소비측이 재구현·미참조.** "공유 자산이 실제로 공유되는지" 역검증. = minor.

---

## 판정 기준

- **CLEAN** — major 신호 0. info/minor만.
- **CONCERN** — major 신호 ≥1, 그러나 추상화 골격은 유지 가능(국소 수정).
- **REDESIGN** — seam/neutrality/isp가 근본적으로 틀어져 추상화를 다시 그려야 함.

---

## 사용법

`abstraction-critic` 에이전트가 대상 추상화(설계 초안 또는 기존 모듈의 공개 표면)를 받아 이 rubric으로 채점하고 findings(JSON)를 낸다. 축 1(commonality)의 서버교차 증거는 `decision-researcher`가 보조한다. 평가 실행·태그 결정은 ops-pilot 몫(자가 채점으로 release tag 결정 금지).
