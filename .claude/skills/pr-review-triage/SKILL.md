# PR Review Triage Skill — 리뷰 읽고 판단·제안

> agent-crew 공유 자산. 프로젝트 고유 값은 `.claude/project.yaml`에서 읽는다.

## 작업 전 — 프로젝트 설정 로드 (선택)

`.claude/project.yaml`이 있으면 Read해 `project.name`·`git`·`project.stack`을 확인한다.
- `project.stack`이 있고 해당 `references/<stack>/` 팩이 있으면 판단 단계에서 컨벤션 근거로 참고한다.
- 없으면 멈추지 말고 gh 기본값·레포 컨벤션으로 폴백한다(프로젝트 비종속).

레포의 `CLAUDE.md`·컨벤션 문서가 있으면 판단 기준으로 같이 읽는다 — 리뷰가 프로젝트 규칙과
충돌하는지 보려면 그 규칙을 알아야 한다.

## 역할

PR에 달린 리뷰를 사람이 매번 직접 읽고 판단하는 수고를, **읽고 분류·제안까지** 대신
해주는 진입점이다. 단순 요약이 아니다 — 각 지적을 **실제 코드와 대조해 타당성을 검증**하고,
수용할 것·반박할 것·보류할 것으로 나눠 우선순위 잡힌 액션 플랜을 내놓는다.

핵심 원칙: **받은 리뷰를 검증 없이 수용하지 않는다.** 리뷰어도 틀릴 수 있고, 트레이드오프나
프로젝트 컨텍스트를 모를 수 있다. 무비판적 동의는 잘못된 수정으로 이어진다. (소비 환경에
`superpowers:receiving-code-review` 스킬이 있으면 그 규율을 따른다 — 없어도 이 원칙은 동일.)

이 스킬은 **읽고 판단·제안까지만** 한다. 코드 수정·PR 답글·resolve는 하지 않는다 —
그래서 도구도 read-only(Bash·Read·Glob·Grep)로 제한했다. 적용은 사람 승인 후 별도 단계다.

## 흐름

### 1. 대상 PR 확정

- 인자로 PR 번호가 오면(`/pr-review-triage 42`) 그 PR을 본다.
- 인자가 없으면 현재 브랜치의 PR을 자동 탐지한다:
  ```bash
  gh pr view --json number,title,headRefName,url,state
  ```
  현재 브랜치에 PR이 없으면 사용자에게 알리고 멈춘다(추측해서 다른 PR을 고르지 않는다).

### 2. 리뷰 수집 (read-only)

리뷰는 세 층위로 흩어진다 — 모두 모은다:

- **리뷰 본문(요약)** — 승인·변경요청과 함께 달리는 총평:
  ```bash
  gh pr view <pr> --json reviews
  ```
- **인라인 코멘트(파일·라인 단위)** — 실제 지적의 대부분이 여기 있다:
  ```bash
  gh api --paginate repos/:owner/:repo/pulls/<pr>/comments
  ```
- **대화 코멘트** — 스레드 일반 코멘트:
  ```bash
  gh pr view <pr> --json comments
  ```

코멘트마다 **파일·라인·작성자(봇/사람)·해결 여부**를 메타로 보존한다. 이미 resolve된
스레드는 우선순위를 낮춘다(필요하면 GraphQL `isResolved`로 구분).

### 3. 판단 — 코드와 대조

코멘트를 하나씩, 해당 파일·라인을 **직접 Read**해 맥락 안에서 평가한다. 요약만 읽고
판단하지 않는다. 평가 축:

- 지적이 **사실인가** — 코드를 보면 정말 그러한가, 리뷰어가 맥락을 놓쳤나
- 프로젝트 **컨벤션·CLAUDE.md와 맞는가** — 규칙과 충돌하는 제안은 아닌가
- **트레이드오프** — 맞는 지적이라도 비용이 이득보다 큰가
- 봇 지적이면 **오탐 가능성** — 정적 분석 false positive인지 코드로 확인

### 4. 분류

각 코멘트를 네 가지로 나눈다:

- ✅ **수용** — 타당하다. *어떻게* 고칠지 + 대상 파일/라인 + 대략의 규모
- 🔁 **반론** — 틀렸거나 트레이드오프가 안 맞는다. *근거와 함께* 반박 초안(리뷰어에게 답할 문장)
- ⏸ **보류** — 범위 밖·추가 정보 필요·후속 이슈감. 왜 지금이 아닌지
- ☑️ **이미 해결/무관** — 닫아도 되는 것(이미 반영됐거나 outdated)

확신이 없으면 추측해서 ✅/🔁로 밀지 말고, 무엇이 불확실한지 명시해 사용자에게 판단을 넘긴다.

### 5. 액션 플랜 제시

분류 결과를 **우선순위·묶음**으로 정리해 터미널 표로 출력한다. 예:

| # | 분류 | 파일:라인 | 지적 요약 | 권장 대응 |
|---|---|---|---|---|
| 1 | ✅ 수용 | `Foo.java:42` | null 체크 누락 | 가드 추가 (소규모) |
| 2 | 🔁 반론 | `Bar.java:88` | "이 캐시 빼라" | 동시성 때문에 필요 — 반박 초안 첨부 |
| 3 | ⏸ 보류 | `Baz.java:10` | 네이밍 전면 변경 | 범위 밖, 후속 이슈 제안 |

표 뒤에 ✅ 수용 항목의 수정 스케치와 🔁 반론 항목의 답글 초안을 덧붙인다.
사용자가 "1·4 고쳐", "2 반박 그대로 달아" 식으로 고르게 한다.

## HITL 경계

- **에이전트가 직접 (read-only)**: PR·리뷰 조회(gh) · 코드 Read·Grep · 판단·분류·플랜·초안 작성
- **사람 승인 후 (이 스킬 밖)**: 코드 수정 · PR 답글/리뷰 코멘트 게시 · 스레드 resolve · push

스킬은 자동으로 코드를 고치거나 PR에 글을 달지 않는다. 승인된 항목의 적용은 일반 편집·답글
워크플로로 이어서 한다.

### Bash 사용 경계 (read-only 강제)

`allowed-tools`에 Bash가 있는 건 `gh`·`git` **조회 전용** 때문이다 (Bash 자체는 쓰기도 가능하므로
read-only 는 도구가 아닌 이 규칙으로 보장한다). 다음만 허용한다:

- `gh pr view`·`gh pr list`·`gh api`의 **GET(조회)** 엔드포인트
- `git log`·`git show`·`git diff` 등 **읽기** 명령

금지: PR 답글·리뷰 게시(`gh pr comment`·`gh api -X POST/PATCH/PUT/DELETE`)·스레드 resolve·
`git commit`·`git push`·파일 수정. 이들은 사람 승인 후 이 스킬 밖에서 한다.

## 산출물

- PR 리뷰 코멘트 전체를 ✅/🔁/⏸/☑️로 분류한 액션 플랜 표
- ✅ 수용 항목별 수정 방향·대상 위치
- 🔁 반론 항목별 근거 있는 답글 초안

## 주의

- **맹목 수용 금지** — 리뷰를 그대로 받아 적지 않는다. 코드를 보고 맞는지 확인한 뒤 분류한다.
- **추측 금지** — 불확실하면 ✅/🔁로 밀지 말고 불확실성을 드러내 사용자에게 넘긴다.
- **수정·답글은 하지 않는다** — 읽고 판단·제안까지가 이 스킬의 전부다(HITL).
- 봇·사람 리뷰를 메타로 구분하되, 봇이라고 무시하거나 사람이라고 무조건 수용하지 않는다 —
  판단 기준은 작성자가 아니라 코드다.
